## 0. 前提（=確定済み・CEO裁定含む）

- 収益モデル: **無料 + プレミアムサブスク**。
- **MVPプレミアム = 広告削除 ＋ 保管枠増加 ＋ 限定Mofi ＋ プレミアム卵**（CEO裁定 2026-06-19）。
- 詳細分析（曜日/時間帯/SNS別/月次/予測）は **v1.1送り**。無料の「今日/今週」分析はMVPに含む（全プラン提供）。
- 通貨: ポイント（pt・卵育成・無料）＋ ジェム（プレミアム通貨。無料入手はマイルストーン報酬限定 / PRD S7）。
- 配信: **Androidファースト → iOS追従**。決済は**ストアIAP（=アプリ内課金）を RevenueCat 経由**で実装。
- 市場: 日本（iOSの課金率が世界的に高め。価格はJPYを基準に設計）。
- App Store Connect / Google Play Console: **法人名義で取得済み**（ORG_STATE）。

## 1. 価格設計（=価格表・おすすめ・トライアル）

### 1-1. 価格表（実額・3.1.2 準拠の表示前提）

| プラン | 実請求額（=主役表示） | 月あたり換算（=従属表示） | 割引 | おすすめ |
|---|---|---|---|---|
| **無料** | ¥0 | — | — | — |
| **プレミアム（月額）** | **¥480 / 月** | — | — | |
| **プレミアム（年額）** | **¥4,800 / 年** | （月あたり ¥400） | **約17%OFF** | ⭐ **おすすめ** |

- **実額最優先**: 年額は「¥4,800/年」を最も目立たせる。「月あたり¥400」は小さく従属表示（=Apple Guideline 3.1.2c / 誤認防止。逆にするとリジェクト直結）。
- 課金画面に必須表記: 「7日間無料、その後 自動更新」「いつでもキャンセル可能」「解約は[設定]→サブスクリプションから」。

### 1-2. ¥480/月の妥当性（=なぜこの価格か）

- **競合アンカー**: 習慣化/スクリーンタイム系（Forest 約¥240買い切り、Finch のサブスク 約¥600〜700/月、Opal 等は$5〜10/月＝¥800〜1,500/月）。Moffyは収集ゲーム性を主軸にした娯楽寄りで、**¥480/月は「娯楽サブスクとしては安い・習慣化アプリとしては中位」**の納得感あるゾーン。
- **日本のモバイルゲーム課金感覚**: 月額480円は「ガチャ1回〜2回分」「コーヒー1杯」の心理的ハードル帯。収集ゲームのライト課金として違和感がない。
- **価値の根拠**: 広告削除＋保管枠20→200＋限定Mofi＋プレミアム卵（高レア卵）の4点パック。1つ1つは小さくても「コレクションを本気でやる人」には月480円の価値がある束。
- **結論**: ¥480/月は**据え置きで妥当**。値上げ余地は残しつつ、初期普及を優先。

### 1-3. 年額 ¥4,800/年 の根拠

- 月額×12 = **5,760円** が基準。割引レンジ 15〜20%。
- ¥4,800/年 = 5,760円から **約17%OFF**（レンジ中央寄り）。月あたり換算**¥400**で「年額にすると実質2.4ヶ月分お得」と訴求できる。
- 末尾を心理価格（4,800の切りのよさ）に丸めて訴求力と暗算しやすさを確保。
- **アンカリング効果**: 月額¥480の隣に「年額¥4,800（月あたり¥400・17%OFF）」を「おすすめ」で置くことで、月額が割高に見え年額へ誘導（=LTV〈顧客生涯価値〉が伸びる年額契約を増やす）。

### 1-4. 無料トライアル: **7日・全商品/全地域に付与（要・採用）**

- 収集ゲームはコアループ（卵→孵化→図鑑）を体験させてからの転換が効く。7日あれば初回孵化〜数体収集を体験でき、プレミアム卵/限定Mofiの魅力を実感させられる。
- 全商品（月額/年額）・全地域に同一付与（オファー機能で設定）。
- 文言は「7日間無料、その後 ¥480/月（または ¥4,800/年）で自動更新」と**実課金額を明示**（3.1.2準拠）。

> **宣伝する値＝実装値（一致原則）**。下表の数値はすべて `lib/core/constants/pricing.dart` の定数を参照。ストア説明・課金画面・保管UIガード・サーバー検証は同じ定数を見る。

### 境界設計の根拠
- **育成枠はプラン非依存**: PRD S6 で「アクティブ卵1枠にのみ加点」と確定済み。育成枠を有料差別化に使うとコアループの公平性が崩れるため、差別化は**保管枠**で行う。
- **保管枠 20→200**: 無料20は「体験は十分・本気で集め始めると足りなくなる」線（pricing-design 定石の Free 設計）。プレミアム200は「ほぼ無制限」の体感を出しつつ、DB肥大・不正対策のため有限上限を持つ（具体数を謳う場合は200に一致させる）。**⚠️監査(2026-07-07): 現状の孵化スループット（育成1枠・週2〜3個消化）＋卵入手レート未定義により、無料ユーザーは20に到達しにくく、保管枠は課金の主レバーになりにくい。主レバーは限定Mofi＋プレミアム卵に置き、保管枠は“補助”と位置づける。数値は卵入手レート確定（PRD §4-4補足）＋公開後 PostHog 実測で再校正する。PRD の旧「無制限」記述は本表（上限あり）に統一済み。**
- **限定Mofi＝課金の主レバー（確定 2026-07-07・財務/グロース合議）**: **MVPでは“種の封鎖”（無料0%の個体）は作らない**。図鑑40は全て無料で到達可能に保ち、「いつかは全部集まる」希望＝収集の根幹満足を壊さない。プレミアムの価値は**「到達“速度”の差」**で作る：無料の標準卵ではSSR3種＋その色違い（計6エントリー）が実質数年がかりなのに対し、**プレミアム卵はSSR排出を約7.6倍速**にする。→ **課金の主レバー＝プレミアム卵**（SSR/色違いへの唯一の現実的加速）＋広告削除、**保管枠は補助**。**追加アート・DB改修なしで成立**（既存40で完結）。⚠️**ストア文言は「40種類のMofiを集めよう」に留め、「必ず全部無料で集まる」と読める表現・色違いを“集めて当然”と煽る表現は避ける（景表法・法務チェック必須）。** ⬜v1.1候補: 天井（プレミアム卵N個でSSR確定等の下限保証／射幸性クレーム対策）／保管済み色違い不採用案を使った「プレミアム限定の専用種・色違い」を数体増設（＝図鑑分母拡張＝アート＋DB改修を伴うので後日・グロース部署案）。
- **広告**: **AdMob バナー広告を実装済み**（`core/ads`・無料プランのみ画面下部に表示／プレミアムは非表示＝「広告削除」の実体／Web・デスクトップは条件付き import で自動 no-op）。現在は Google 公式テスト広告ID（`core/ads/ad_config.dart`）で、本番収益化時に AdMob で作成した実ユニット/アプリIDへ差し替える（AndroidManifest.xml / ios Info.plist のアプリID含む）。**実装＝宣伝の一致**が取れたので「広告削除」を有料特典として正当に訴求できる。⚠️検証は実機（Android APK/iOS）で行う（Web プレビューは AdMob 非対応で広告が出ない）。
- **詳細分析は両プラン false**: v1.1送りのため、課金画面で「詳細分析」を**プレミアム特典として宣伝しない**（=実装されていない機能を売ると景表法・3.1.2の誇大表示リスク。MVPの売りは体験系4点に絞る）。

## 3. ユニットエコノミクス試算（=1ユーザーあたりの採算）

> 実数値未確定のためレンジ。前提は明記。MAU=月間アクティブユーザー、ARPU=1ユーザーあたり平均収益、ARPPU=課金ユーザーあたり平均収益。

### 3-1. 前提（=置いた値と根拠）

| 項目 | 置いた値 | 根拠/出典 |
|---|---|---|
| ストア手数料 | **30%（初期）→ 15%（小規模事業者プログラム承認後）** | Apple Small Business Program / Google「年100万ドル以下は15%」。**早期申請必須**（後述§6） |
| 課金CVR（無料→有料） | **2〜5%**（基準3%） | 一般的なフリーミアムサブスクの課金率。収集ゲーム性で上振れ余地 |
| 月額/年額比率 | 月額60% / 年額40% | 年額をおすすめ表示で誘導した想定 |
| プレミアム実効単価/月 | 月額¥480、年額¥4,800÷12=¥400 → 混合 **約¥448/月** | 60:40 加重平均 |
| 変動コスト（バックエンド） | Supabase / Sentry / PostHog | いずれも**無料枠あり**。小規模初期はほぼ固定費（後述） |
| AI推論コスト | **¥0** | MoffyはLLM/AI推論を使わない（抽選はサーバーRPCの確率計算のみ。AIアプリではない） |

### 3-2. 変動コスト/ユーザー（=月次）

- **Supabase**: Free枠（DB 500MB / 認証 5万MAU / 帯域 5GB）→ Pro $25/月（8GB DB・10万MAU）。Moffyは1ユーザーのデータ量が小さい（pt・卵・図鑑・利用統計の軽量行）ため、**数千〜1万MAUまではほぼ固定費**。1ユーザーあたり変動費は **¥1未満/月** のオーダー。
- **Sentry**: 開発者向け無料枠（月5kエラー）。クラッシュ監視は量課金だが安定後はエラー数が減るため小規模では実質固定。
- **PostHog**: 無料枠（月100万イベント）。主要ファネル計測程度なら**当面無料枠内**。
- **結論**: Moffyは**変動コストが極小**（AI推論ゼロ・データ軽量）。コスト構造は「ほぼ固定費（バックエンドのプラン費）」で、**ユーザーが増えるほど1人あたりコストが下がる**良い形。損益分岐は「固定費を粗利で割る」で読める。

### 3-3. 粗利（=プラン価格 ×(1−手数料) − 変動費）

粗利 ≈ 実効単価 × (1 − 手数料)（変動費が極小のため無視できるオーダー）。

| 手数料 | 月額¥480の粗利/月 | 年額¥4,800の粗利/年 | 混合実効 約¥448/月の粗利/月 |
|---|---|---|---|
| **30%**（初期） | ¥336 | ¥3,360 | **約¥314/月** |
| **15%**（承認後） | ¥408 | ¥4,080 | **約¥381/月** |

### 3-4. ARPU と損益分岐の目安

- **ARPU（全ユーザー平均）** = 実効単価 × CVR。
  - CVR=3% × 実効¥448 = **ARPU 約¥13/月（粗利ベースは手数料30%で約¥9、15%で約¥11/MAU）**。
- **損益分岐（固定費回収）**: バックエンド固定費を仮に **¥4,000/月**（Supabase Pro $25＋予備）と置くと、
  - 必要粗利 ¥4,000 ÷ 粗利単価（手数料30%・混合¥314/月）= **約13人の課金者**で固定費回収。
  - CVR3%なら **MAU 約430人** で固定費トントン。これは極めて低いハードル＝**早期に固定費は回収可能**。
- **示唆**: Moffyは「変動費が薄い・固定費が小さい」ため、**赤字リスクは無料枠の暴走ではなく、そもそもの課金者数（CVR×MAU）**にある。財務の主戦場は「いかに課金転換させるか（グロース）」であり、コスト管理は当面軽い。
- **注意点**: 上記は机上前提。実CVR・MAU・年額比率は launch 後 PostHog で実測し更新する（数字メモは ORG_STATE）。

## 4. RevenueCat 商品設計

### 4-1. 構成（products → entitlement → offering）

```
Offering: "default"
  ├─ Package $rc_monthly → Product: moffy_premium_monthly  (¥480/月 + 7日トライアル)
  └─ Package $rc_annual  → Product: moffy_premium_yearly   (¥4,800/年 + 7日トライアル)

Entitlement: "premium"
  ← moffy_premium_monthly / moffy_premium_yearly の両方を attach
```

- 識別子は **`lib/core/constants/pricing.dart` の `RevenueCatIds`** が唯一の定義（クライアント・サーバー・ストア商品登録で同一文字列）。
  - `entitlementPremium = 'premium'` / `defaultOffering = 'default'`
  - `productMonthly = 'moffy_premium_monthly'` / `productYearly = 'moffy_premium_yearly'`
  - `packageMonthly = $rc_monthly` / `packageYearly = $rc_annual`
- プレミアム判定は**常に entitlement `premium` の有効性**で行う（商品IDで分岐しない＝将来の商品追加に強い）。
- 価格・トライアルは **App Store Connect / Play Console 側で設定**（RevenueCatは仲介）。表示価格は `StoreProduct.priceString`（ローカライズ実額）を使い、`pricing.dart` の円価格は特商法表記・比較UI・採算試算の参照に限定。

### 4-2. サーバー検証（entitlements RPC / Webhook）との整合

- **真のSSOTはサーバー**: クライアントの `CustomerInfo` は表示用。プレミアム特典の付与可否（保管枠200の解放・プレミアム卵導線）は**サーバーが entitlement を検証**して決定（不正クライアント対策）。
- **RevenueCat Webhook → Supabase**: 購入/更新/解約/返金イベントを受け、ユーザーの `is_premium`/`premium_until` 等をサーバー側で更新（Edge Function 等）。アプリの保管枠ガード・限定Mofi抽選はこのサーバー値を参照。
- **App User ID 紐づけ**: PRD S10 の匿名認証→連携と整合。RevenueCat App User ID を Supabase の user_id に揃え、機種変時の復元（Restore Purchases）が連携アカウントで効くようにする。
- **既存実装との接続**: ORG_STATE の「課金(RevenueCat→entitlements Webhook)」は未着手。開発引き継ぎ時に、本構成（IDは `RevenueCatIds`）でWebhook受信→サーバー状態更新→クライアントは entitlement 参照、の3層で配線する。

### 4-3. サンドボックス検証の要点

- **Android**: Play Console の**ライセンステスター**登録＋内部テストトラックで実購入フロー（購入/復元/解約/トライアル開始→課金移行）を検証。トライアル7日の自動更新はテスト時間短縮設定で確認。
- **iOS（追従期）**: App Store Connect の**Sandboxテスター**で同様。Sandboxは更新サイクルが短縮（例: 年額が数分で更新）されるので、トライアル→課金移行・解約反映を確認。
- **検証項目**: ①トライアル開始で entitlement `premium` が即有効化 / ②保管枠が200に解放（サーバー検証経由）/ ③解約で更新停止後に失効 / ④返金/期限切れで特典剥奪 / ⑤Restoreで別端末復元（連携アカウント）/ ⑥Webhookイベントがサーバー状態に反映。
- **一致原則の検証**: ストア説明で謳う保管枠数・特典が、実際に解放される値（`pricing.dart`）と一致することを実機で突合（誇大表示・クレーム防止）。

## 5. ジェム設計（=無料入手レート・プレミアムでの位置づけ・インフレ防止）

> PRD S7 準拠。真のSSOTはサーバー（app_config/報酬テーブル）。`GemEconomy`（pricing.dart）は表示・整合確認の参照値。

### 5-1. 無料入手レート（マイルストーン限定）

| 入手源 | レート | SSOT定数 |
|---|---|---|
| ウィークリークエスト達成 | 週あたり最大 **40ジェム** | `GemEconomy.weeklyQuestGemCap = 40` |
| 図鑑マイルストーン（5/10/15/20/25/30体） | 1段ごと **30ジェム**（全6段＝計180） | `dexMilestoneGem=30` / `dexMilestoneCount=6` |
| ストリークマイルストーン | 7日=**20** / 30日=**50** | `streak7DayGem=20` / `streak30DayGem=50` |

- 広告視聴報酬・ログインボーナスでのジェム配布は**しない**（PRD S7。希少性維持）。

### 5-2. プレミアムでの位置づけ・インフレ防止

- ジェムの**主用途はプレミアム卵**（高レア卵 / PRD §4-4：Normalを出さない課金価値担保）。`premiumEggGemCost = 120`。
- **インフレ防止の設計**: 無料の安定供給は週40＋図鑑/ストリーク。プレミアム卵120ジェムは「無料勢で数週に1個」のペース。プレミアム購読者は「広告削除＋保管枠＋限定Mofi＋プレミアム卵導線」のパッケージで体験を加速するが、**ジェム自体を大量配布するサブスクにはしない**（=ジェムの希少性が崩れると課金動機・収集の達成感が消える）。
- プレミアムは「ジェムをばらまく」のではなく「**ジェムで引けるプレミアム卵という到達点を解放し、限定Mofiという出口を用意する**」設計。無料勢にも到達経路を残しつつ、プレミアムで体験が滑らかになる。

## 6. 手数料優遇プログラム（=早期申請事項）

- **Apple Small Business Program**: 年間売上100万ドル以下なら手数料 30%→15%。**承認制・年次登録**。launch 前に申請（承認に時間がかかるため）。承認後は RevenueCat / ASC 側の適用開始日を反映。
- **Google Play**: 各開発者アカウントの年間最初の100万ドルは自動で15%（申請不要だが条件確認）。

# Moffy PRD（=プロダクト要求仕様書 / Product Requirements Document）v1.0

> 作成: 企画部署（product-planner） / 日付: 2026-06-19
> 位置づけ: 要件定義書 v1.0 の「不足仕様 S1〜S14」を**部署として決め切った**確定版。デザイン（画面化）・開発（実装）はこの受け入れ条件を満たせば「完成」とみなす。
> 専門用語は初出時にインラインで日本語説明。

---

## 1. 一文定義

**「SNSを見すぎてしまう18〜35歳」が「スマホ時間を減らせない課題」を「削減した時間でかわいいキャラ(Mofi)が育つ収集ゲーム化」で解決する。**

- コア価値はただ1つ: **削減した分だけ卵が育ち、孵化して図鑑が埋まる**という1本のループ。これ以外は「レール（=共通基盤）」か「v1.1+送り」。
- SNS断ちが目的ではない。**収集欲を燃料に、結果として利用時間が減る**設計。

---

## 2. コアループ（=v1.0 で完璧に磨く唯一の体験）

```
アプリ起動
  ↓
本日のSNS利用時間を確認（OSのスクリーンタイムから自動取得）
  ↓
基準値（過去7日平均）より削減できた分 = ポイント（pt）
  ↓
育成中の卵がポイントで成長（100pt→ヒビ① / 250pt→ヒビ② / 500pt→孵化）
  ↓
孵化演出 → Mofi獲得（卵レアリティ→Mofiレアリティ→種族→色違いの順で抽選）
  ↓
図鑑登録（達成率が上がる）
  ↓
新しい卵を育成枠にセット → 繰り返し
```

補助ループ: デイリー/ウィークリークエスト達成 → 報酬（ポイント/卵/ジェム） → ストリーク（連続達成）でポイント倍率アップ。

## 3. 確定仕様（S1〜S14：決定と理由）

> 凡例: 各項目は **【決定】**（=実装する仕様）、**【理由】**、必要に応じ **【やらない】**（=v1.1+送り）を記載。

### S1 基準値の初日問題（7日平均が無い新規ユーザー）

- **【決定】** 段階的に基準を確定する **「ウォームアップ方式」** を採用。
  - **Day 1〜2（データ0〜1日）**: 基準値は**まだ計算しない**。代わりに「**初回ボーナス卵**」を1個付与し、孵化までを**固定ポイント（合計500pt相当を3日に分けて自動付与: Day1=200pt, Day2=300pt）**で体験させる。実利用時間に依存しない「お試し孵化」でコアループを最短で体験させる。
  - **Day 3〜6（データ2〜5日）**: 「**暫定基準値 = それまでに取得できた日の平均**」を採用。表示に「基準値は7日分たまると確定します（暫定）」と明示。
  - **Day 7 以降**: 正式に **過去7日平均**を基準値に確定。
- **【決定】基準値の下限ガード**: 基準値が **30分未満**になった場合は基準値を**30分でクランプ（=下限固定）**。理由は S2 参照（既に低利用の優等生が0ptで詰むのを防ぐ）。
- **【理由】** 「初日に何も起きない」が最大の離脱要因。7日待たせるのは致命的。ウォームアップでD1孵化を保証し、コアループの快感を即体験させる。暫定基準は後で正式値に置き換わるが、初期の体験を阻害しない。
- **【受け入れ条件は §5 に集約】**

### S2 利用が増えた日（削減量マイナス）の扱い

- **【決定】** 削減量がマイナス（=基準より使ってしまった日）でも**ポイントはマイナスにしない（下限0pt）**。卵の成長ポイントは**減算しない（不可逆・蓄積型）**。
- **【決定】** ただしマイナス日は **ストリーク（連続達成）を途切れさせる**（=デイリークエスト未達と同様の扱い、S14参照）。
- **【決定】** マイナス日のホーム表示は責めない文言にする（例「今日は少し多めだったみたい。明日はMofiのために少し減らそう」）。
- **【理由】** ペナルティで卵が縮むと「育てた時間が奪われる」喪失感が強く、離脱と星1レビューに直結。Finch/習慣化アプリの知見では「罰より復帰のしやすさ」が継続率を上げる。倍率や連続性のインセンティブで前向きに動機づける。

### S3 複数SNS合算の定義（対象アプリ合計か全アプリか）

- **【決定】** ポイント計算の対象は **「対象SNSアプリの合計利用時間」** とする。MVPの対象は **TikTok / Instagram / YouTube / X の4つ**（要件どおり）。全アプリ合計（=総スクリーンタイム）は対象外。
- **【決定】** Android実装は `UsageStatsManager`（=アプリ別利用統計を返すOS API）で**対象4アプリのパッケージ名をホワイトリスト化**して合算。リスト外アプリは無視。
- **【決定】対象アプリは設定で増減できる構造**にするが、**MVPでは固定4アプリ**。ユーザーによる対象アプリのカスタマイズはv1.1+。
- **【理由】** 「全アプリ合計」だと地図・仕事・読書アプリの利用まで罰してしまい、アプリの主張（=SNSの見すぎ対策）とズレる。対象を絞ることで「狙ったSNSを減らす」という体験が明確になる。

### S4 ポイント上限/不正対策（OS時刻改ざん等）

- **【決定】1日あたりポイント上限 = 480pt/日**（=8時間削減相当。基準値の現実的上限を超える分は加点しない）。
- **【決定】不正対策（MVPで実装）**:
  1. **OS時刻改ざん検知**: 端末ローカル時刻ではなく、**Supabaseサーバー時刻（=NTP同期されたバックエンドの時刻）を日付境界の正とする**（S11参照）。同期時にサーバー時刻でその日の確定を行う。
  2. **当日分のみ確定**: ポイント確定は「サーバー基準でその日が終了した分」を翌日に確定。**未来日付・過去日付の遡及加点は不可**。
  3. **異常値ガード**: 1日の取得利用時間が物理的にありえない値（例 24時間超）は破棄しログ記録。
  4. **端末ロールバック検知**: 端末時刻がサーバー時刻より大きく巻き戻った（例 24時間以上）場合、当日のポイント確定を保留しサーバー再同期を待つ。
- **【やらない（v1.1+）】** 機械学習による高度な不正検知、複数端末間の同一アカウント不正利用の厳密検知（=MVPは1ユーザー1端末前提で十分）。
- **【理由】** ゲーム内通貨に実害（=リアルマネー流出）がなく、不正してもユーザー自身の達成感を損なうだけなので、過剰防御は不要。ただしランキング等の将来機能の土台として、**日付境界はサーバー時刻**という原則だけは最初から固める。

### S5 孵化時のMofi抽選ロジック（卵レアリティ→Mofiレアリティ→種族→色違い）

詳細な確率表は **§4** に集約。抽選順序は以下。

- **【決定】抽選順序**: ①卵レアリティは**入手時に確定済み**（孵化時は再抽選しない）→ ②そのレア卵に紐づく**Mofiレアリティ分布表でレアリティを抽選** → ③そのレアリティに属する**種族・個体を均等抽選** → ④**色違い判定**（S13参照）。
- **【決定】MVPの全Mofi構成**: 4種族（スライム/小動物/ドラゴン/獣）× 各5種 = **20種**。各種に**通常色 + 色違い**があるので図鑑総数は **40エントリー**。（獣は男性訴求の“かっこいい”枠として2026-07-02追加）
- **【決定】レアリティと種族の対応はMVPでは独立**（=どの種族にも各レアリティ個体がいるわけではなく、§4の個体マスタで「この個体はSR」と固定）。抽選は「レアリティを引く→そのレアリティの個体プールから均等に1体」。
- **【理由】** 4段階の入れ子抽選を明示しないと、開発が確率実装を自己流で組んでバランスが崩れる。順序と確率を数値で固定し、後からサーバー側パラメータで調整可能にする（=確率はSupabaseのリモート設定で変更できる構造）。

### S6 育成枠(3枠)が埋まった時の挙動

- **【決定】** 育成枠は3枠固定。**ポイントは「アクティブな育成卵1枠」にのみ加算**する（=同時並行育成はしない）。3枠は「育成中の卵を切り替えられるスロット」であり、ユーザーが**任意のタイミングでアクティブ卵を切り替え**できる。
- **【決定】3枠が埋まっていて新しい卵を入れたい**場合: 保管枠から育成枠への入れ替えは、**育成枠の卵を保管枠に戻す**ことでいつでも可能（成長ポイントは卵ごとに保持され、戻しても失われない）。**保管枠は上限あり＝無料20/プレミアム200（SSOTは `lib/core/constants/pricing.dart`。旧記述「無制限」は誤り→本値に統一・宣伝＝実装の一致原則）。** ただし監査(2026-07-07)により保管枠は課金の主レバーにはなりにくい見込み（→§4-4補足）。主レバーは限定Mofi＋プレミアム卵。
- **【決定】孵化して空いた枠**には、保管枠の卵を手動でセット（=自動補充しない。ユーザーに「次にどれを育てるか」を選ばせる方が収集の意思決定が楽しい）。育成枠が全部空でアクティブ卵が無い間に得たポイントは**最大3日分プール**し、卵セット時に充当（取りこぼし防止）。
- **【理由】** 「3枠同時にポイント山分け」は計算が複雑でユーザーに伝わらない。「今日育てる1個」を明確にする方が、毎日の削減と卵の成長の因果が分かりやすい。3枠は「気分で育てる卵を変えられる」収集の遊び。

### S7 ジェム（=プレミアム通貨）の無料入手手段

- **【決定】無料入手手段（MVP）**:
  1. **ウィークリークエスト達成報酬**（=主たる無料入手源。週あたり最大 約30〜50ジェム）。
  2. **図鑑マイルストーン報酬**（例: 図鑑5/10/15/20/25/30体達成ごとにジェム付与）。
  3. **ストリークマイルストーン**（7日/30日連続達成でジェム）。
  4. **アプリ内レビュー依頼に応じた等の初回限定ボーナスは行わない**（=ストア規約で「レビューと報酬の交換」は禁止のため。S付与しない）。
- **【決定】デイリーログインボーナス**は**MVPでは少額のポイント（ジェムではない）**に留める。ジェムは「やや貴重」を保つ。
- **【決定】広告視聴によるジェム入手は MVPではやらない**（=広告SDK未導入。v1.1+で検討）。
- **【理由】** ジェムが無料でジャブジャブ手に入るとプレミアム購入動機が消える。一方で完全に課金専用だと収集の達成感が薄れる。「クエストと図鑑達成のご褒美」に限定し、希少性と達成感を両立。

### S8 オフライン→オンライン競合解決ルール

- **【決定】SSOT（=Single Source of Truth / 信頼できる唯一の情報源）の原則**:
  - **利用時間の生データ**: 端末ローカル（Drift）がSSOT。OSから取った値は端末でしか取れないため。
  - **ポイント確定・卵成長・図鑑・通貨残高**: **サーバー（Supabase）がSSOT**。
- **【決定】競合解決ルール**:
  1. オフライン中は端末で**暫定計算**してUI即時反映（=楽観的更新）。
  2. オンライン復帰時、端末の「未確定の利用時間生データ」をサーバーに送信し、**サーバーが日付ごとに確定計算**して正とする。
  3. 同一日が端末とサーバーで食い違う場合、**サーバーの確定値を採用**（=ポイント・成長は上書き）。ただし**確定済みポイントを減らす方向の上書きはしない**（=ユーザー保護。増える方向のみ反映、減る差分はログのみ）。
  4. 通貨の消費（ジェム/ポイント使用）は**サーバーで原子的に処理**し、オフライン中の消費は許可しない（=二重消費防止。残高不足UIを出す）。
- **【理由】** ゲーム経済の不整合（=通貨二重取得）はユーザー信頼を壊す。「生データは端末・確定は鯖」と役割を分けることで、オフライン体験を保ちつつ整合性を担保。

### S9 通知の種類とトリガー

- **【決定】MVPの通知（ローカル通知中心、プッシュは最小限）**:
  | 通知 | トリガー | 既定 |
  |---|---|---|
  | デイリーリマインド | 毎日設定時刻（既定21:00、ユーザー変更可） | ON |
  | 孵化間近 | 育成中の卵が孵化まで残り少ない（例 残50pt以内）と判定時 | ON |
  | ストリーク途切れ警告 | その日のデイリー未達かつ就寝前（既定22:00）に未達なら | ON |
  | クエスト達成可能 | 達成条件をほぼ満たした時 | ON |
  | 週次サマリー | 週末（日曜夜）に「今週の削減時間とMofi獲得」 | ON |
- **【決定】** すべて**設定で個別ON/OFF可能**。OS通知許可は**コアループを1回体験させた後**に依頼（=許可率向上）。
- **【決定】** MVPは**ローカル通知（=端末内スケジュール）**を基本とし、サーバープッシュ（FCM）は**週次サマリー等の将来拡張用に構造だけ用意**し、MVPでは未使用でも可。
- **【理由】** 継続率は通知設計で大きく変わる（Duolingoの知見）。ただし通知過多は即アンインストール要因なので、初期は控えめ・全部OFF可能に。

### S10 アカウント方式（Supabase Auth）

- **【決定】「匿名認証ファースト → 任意で連携」** を採用。
  1. **初回起動: Supabase匿名認証（Anonymous Sign-In）で即座にアカウント作成**。サインアップ画面を挟まず、すぐコアループに入れる。
  2. データはこの匿名アカウントに紐づきサーバー保存（=機種変・再インストール対策のため、後で連携を促す）。
  3. **任意のタイミングで「アカウントを引き継ぐ」導線**を提示（例: 図鑑がたまってきた頃、プレミアム購入時）。連携方式は **Sign in with Apple / Google / メールマジックリンク**。
  4. プレミアム購入（サブスク）は**連携アカウント推奨**だが、RevenueCatの匿名IDで紐づくため匿名のままでも購入は技術的に可能。ただし**機種変での復元には連携が必要**と明示。
- **【決定】** **Sign in with Apple は iOS で必須**（=他社ソーシャルログインを出す場合 Appleの提供が審査要件 4.8/5.1）。Androidファースト段階では Google + メールで開始し、iOS追従時に Apple を追加。
- **【理由】** サインアップ障壁はオンボーディング最大の離脱点。匿名で即開始 → 価値を感じた後に連携、が収集ゲームの定石。ただし「機種変でMofiが全部消えた」が最悪の星1レビューなので、連携導線とリスク明示は必須。

### S11 タイムゾーン/日付境界（「本日」「7日」の確定基準）

- **【決定】日付境界 = ユーザーのローカルタイムゾーンの 0:00〜23:59**（=「本日」は端末のローカル日付）。生活実感に合わせる。
- **【決定】確定の正は「サーバー時刻 + ユーザー登録タイムゾーン」**: ユーザーのタイムゾーンを**アカウントに保存**し、サーバーはそのTZで日付を区切って確定計算する（S4の改ざん対策と両立）。
- **【決定】「過去7日平均」の定義**: **本日を含まない直近7日間（=昨日からさかのぼって7日）**の対象SNS合計利用時間の平均。データ欠損日（=取得できなかった日）は平均計算から除外（=分母を実データ日数にする）。
- **【決定】タイムゾーン変更時**（=旅行・移住）: ユーザーTZ変更を検知したら、**変更日は二重カウント/欠損が起きないようサーバーで調整**（=変更日は当日のみ再計算、過去は確定済みを維持）。MVPでは「TZ変更は稀」前提で、当日処理のみ厳密化。
- **【理由】** 「本日」がブレると削減量計算が崩れ、ユーザーは不信感を持つ。ローカルTZで生活実感に合わせつつ、確定はサーバーTZ管理で改ざん耐性を持たせる。

### S12 退会・データ削除（審査必須）

- **【決定】アプリ内に「アカウント削除」導線を必ず設置**（=Apple 5.1.1(v) / Google のデータ削除要件で必須）。プロフィール/設定画面からアクセス。
- **【決定】削除フロー**:
  1. 「アカウント削除」→ 影響説明（=Mofi・図鑑・購入履歴がすべて消える、復元不可）→ 確認 → 実行。
  2. 実行で **Supabase上のユーザーデータを削除**（=auth user, ポイント, 卵, 図鑑, 利用統計, プロフィール）。Drift（端末）データもクリア。
  3. **サブスク契約の扱いを明示**: 「アカウント削除してもストアのサブスク契約は自動解約されない。解約は各ストアの定期購読管理から」と案内（=App Store/Google Playの仕様。アプリ側で解約できないため）。
  4. **アプリ外（Web）からの削除リクエスト窓口**も用意（=mailto または問い合わせフォーム。Google要件で「アプリを再インストールせずに削除依頼できる手段」が求められる）。
- **【決定】削除完了までの期限**: 即時論理削除 + **30日以内に物理削除**（=バックアップ含む）。プライバシーポリシーに保持期間を明記。
- **【理由】** これは「機能」ではなく**審査を通すための必須レール**。導線が無い/動かないと確実にリジェクト。法務部署のプライバシーポリシーと整合させる。

### S13 色違い出現率

- **【決定】色違い出現率 = 一律 1/50（=2.0%）**。Mofiレアリティ・種族に関わらず、抽選で個体が決まった後に独立判定。
- **【決定】色違いは別図鑑エントリー**（=図鑑総数30に含む。通常色15 + 色違い15）。図鑑コンプ率に反映。
- **【決定】色違いの孵化演出は専用エフェクト**（=「キラリ」演出。SNS共有したくなる瞬間を作る = グロースの種）。
- **【理由】** ポケモン等の色違い文化に倣い、「2%」は十分レアで価値があり、かつ数十〜百体引けば出会える希望のある確率。レア演出はオーガニックなシェア（=口コミ拡散）の核になる。

### S14 ストリーク倍率の適用対象（ポイントのみか報酬全般か）

- **【決定】ストリーク倍率は「削減で得る基礎ポイント」にのみ適用**。
  - 対象: 1分削減=1ptで計算した**その日の基礎ポイント**に倍率（×1.0〜×2.0）を掛ける。
  - **対象外**: クエスト報酬ポイント、ジェム、卵などの**固定報酬には倍率を掛けない**。
- **【決定】倍率テーブル**: 連続達成日数で 1日=×1.0 / 3日=×1.2 / 7日=×1.5 / 30日=×2.0（要件どおり）。間の日数はその直下の段を適用（例 5日目は×1.2、10日目は×1.5）。
- **【決定】ストリーク継続条件**: 「その日のデイリークエストを1つ以上達成」または「その日の削減量がプラス（基準より減らせた）」のいずれか。マイナス日・未達日でストリークはリセット（S2参照）。
- **【決定】ストリーク倍率適用後も S4 の1日上限480ptは維持**（=上限は倍率適用後の最終値に対して判定）。
- **【理由】** ジェムや卵にまで倍率が乗ると経済が壊れ、プレミアム動機も薄れる。「毎日削減する」というコア行動への報酬だけを増幅させ、習慣化を最大化する。

## 5. 受け入れ条件（=この条件を満たせば各機能は「完成」）

> 各主要機能に **5状態（ハッピーパス / エラー / ローディング / 空状態 / オフライン）** を必須化。

### 5-1. 利用時間取得とポイント計算

- [ ] **ハッピーパス**: Androidで対象4アプリの利用時間を `UsageStatsManager` から取得し、基準値（7日平均）との差分をptとして算出・表示できる。
- [ ] **初日（S1）**: データ0〜1日のユーザーに初回ボーナス卵が付与され、Day1で孵化体験まで到達できる。Day3〜6は暫定基準で「暫定」表示が出る。
- [ ] **マイナス日（S2）**: 削減量マイナスでもptは0止まり、卵成長は減らない。責めない文言が表示される。
- [ ] **上限（S4）**: 1日480ptを超えて加算されない。
- [ ] **エラー**: OS利用統計の権限が無い/取得失敗時、権限再要求の導線とリトライを表示。
- [ ] **ローディング**: 取得・同期中はスケルトン/スピナー表示。
- [ ] **空状態**: 対象アプリの利用が0分の日も正しく処理（削減量=基準値、上限内で加算）。
- [ ] **オフライン**: 端末で暫定計算しUI反映、オンライン復帰でサーバー確定値に整合（S8。減る方向の上書きはしない）。

### 5-2. 卵の育成・孵化・抽選

- [ ] **ハッピーパス**: アクティブ卵にptが累積し、100/250/500ptで見た目が変化、500ptで孵化演出 → §4の確率でMofi確定 → 図鑑登録。
- [ ] **抽選（S5/S13）**: §4の確率表どおりにレアリティ→個体→色違いが抽選される（=確率は単体テストで分布検証: 大量試行で理論値±許容誤差内）。
- [ ] **3枠（S6）**: アクティブ卵切り替え、育成枠↔保管枠の入れ替えで成長ptが保持される。空枠で得たptは最大3日プールされ次卵に充当。
- [ ] **エラー**: 孵化処理がサーバーで失敗した場合、ptを消費せずリトライ可能（=二重孵化しない、原子的処理）。
- [ ] **ローディング/演出**: 孵化演出中に操作ロック、演出スキップ可。色違いは専用演出。
- [ ] **空状態**: 育成枠が全て空の時「卵をセットしよう」の誘導と保管卵一覧表示。
- [ ] **オフライン**: 孵化はオンライン確定が原則。オフライン中は「孵化準備完了・接続時に確定」表示。

### 5-3. クエスト・ストリーク・通貨

- [ ] **ハッピーパス**: デイリー/ウィークリークエストが自動生成・進捗反映・達成で報酬付与。ストリーク倍率が基礎ptにのみ適用（S14）。ジェムが§S7の手段で入手できる。
- [ ] **エラー**: 報酬付与がサーバーで失敗時、二重付与せずリトライ。
- [ ] **ローディング/空状態/オフライン**: 各々スケルトン・「今日のクエスト準備中」・オフライン時は進捗ローカル保持→復帰時整合。

### 5-4. アカウント・データ管理（S10/S12）

- [ ] **ハッピーパス**: 初回起動で匿名認証され、サインアップ無しでコアループに入れる。任意で Apple(iOS)/Google/メール連携できる。
- [ ] **退会（S12）**: 設定からアカウント削除でき、サーバー/端末データが消え、サブスク解約の案内が表示される。アプリ外削除窓口（mailto/フォーム）が存在する。
- [ ] **エラー/オフライン**: 連携・削除はオンライン必須。オフライン時は導線をグレーアウトし理由表示。

### 5-5. 通知（S9）

- [ ] 5種の通知が設定で個別ON/OFFでき、初回はコアループ1回後に許可依頼。各トリガーで正しく発火。

### 5-6. 標準レール（MVP必須・機能数に数えない）

- [ ] **課金**: RevenueCatでプレミアムサブスク(¥480/月)の購入・復元・解約導線・サブスク条件表示（3.1.2準拠）。
- [ ] **アプリ内レビュー依頼**: コアの達成体験後（例: 初回孵化や図鑑5体）に `in_app_review` を1回トリガー。
- [ ] **フィードバック導線**: 設定にmailto（最低限）。
- [ ] **行動分析（PostHog）**: 主要ファネル（起動→利用時間取得→pt獲得→孵化→図鑑登録→課金）のイベントが初日から計測される。
- [ ] **クラッシュ監視（Sentry）**: クラッシュ・主要エラーが捕捉される。
- [ ] **法務文書**: プライバシーポリシー/利用規約/特商法表記がアプリ内から閲覧でき、プライバシー申告と整合（法務部署成果物とリンク）。

## 6. やらないこと（=v1.1+ 送り。スコープクリープ防止）

> 「コア価値1つを完璧に」のため、以下はMVPでは**実装しない**と明記する。

- **iOS版（v1.1）**: Androidで経済を実数値検証後に追従。iOSは段階しきい値近似（ORG決定済み）。`FamilyControls` Entitlement申請のみ早期着手。
- **ガチャ / イベント / 期間限定Mofi / シーズンパス**: 抽選は卵孵化に内包。直接ガチャは将来。
- **ランキング / フレンド / ギルド / 交換機能**: ソーシャル機能全般。
- **着せ替え / スキン**: ジェム用途として将来。MVPはジェムの用途を「プレミアム卵」中心に絞る。
- **対象SNSアプリのユーザーカスタマイズ**: MVPは固定4アプリ（S3）。
- **広告SDK・広告視聴報酬**: MVPは広告なし（S7）。プレミアムの「広告削除」は将来の広告導入を見越した文言整理のみ。
- **高度な不正検知（ML・多端末厳密検知）**: MVPはサーバー時刻＋上限＋異常値ガードで十分（S4）。
- **サーバープッシュ通知の全面運用**: MVPはローカル通知中心。FCMは構造のみ用意（S9）。
- **詳細分析の全機能**: 曜日/時間帯/SNS別/月次/予測はプレミアム機能だが、MVPの分析は「今日/今週」の基本に絞り、詳細分析は最小実装 or v1.1で拡充（CEO/財務と要相談）。
- **タイムゾーン変更の厳密な全期間再計算**: MVPは「変更日のみ厳密化」で割り切り（S11）。
